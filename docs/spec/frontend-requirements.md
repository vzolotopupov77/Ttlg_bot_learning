# Требования к веб-интерфейсу (frontend)

Документ фиксирует функциональные требования к экранам и общим компонентам веб-приложения. Ориентир по продукту — [vision.md](../vision.md). Визуальные и поведенческие детали согласованы с референсом [tutor-schedule.lovable.app](https://tutor-schedule.lovable.app/) (скриншоты в ходе итерации 0).

**Стек (зафиксирован в tasklist):** Next.js (App Router), TypeScript, shadcn/ui, Tailwind CSS, pnpm.

---

## Принципы

- **Тонкий клиент:** бизнес-логика и данные — в backend; веб только отображает и отправляет запросы по контрактам ([api-contracts.md](../tech/api-contracts.md), раздел «API для frontend»).
- **Роли:** `teacher` и `student` ([data-model.md](../data-model.md)); доступ к маршрутам по роли.
- **Аутентификация:** без верификации e-mail в MVP; сессия через **JWT в httpOnly cookie** (не в `localStorage`).
- **Темы:** светлая и тёмная; переключатель в шапке; выбор пользователя сохраняется (например `localStorage` + `next-themes`), без мигания при загрузке (FOUC).
- **Адаптивность:** формы и основные экраны читаемы от ~375px ширины; на мобильных боковая панель скрывается за drawer.

---

## Общие компоненты

### Боковая навигация (sidebar)

- **Преподаватель:** пункты «Календарь», «Ученики», «Настройки»; внизу «Выйти».
- **Ученик:** один пункт «Моё расписание»; внизу «Выйти».
- Фон sidebar **тёмный (navy)** в обеих темах приложения; активный пункт визуально выделен.
- Логотип: иконка шапки + текст «Репетитор».

### Верхняя панель (header)

- Заголовок текущей страницы (слева).
- Справа: переключатель темы (луна / солнце) и (где нужно) элементы профиля; на экранах после входа — выход через sidebar или header по решению реализации, но «Выйти» доступен.

### Мобильная навигация

- На узкой ширине: гамбургер открывает **Sheet** (drawer) с тем же меню; обязательны доступность (`SheetTitle`, скрытый заголовок допустим).

## Аутентификация и маршруты

| Роль после входа | Стартовая страница |
|------------------|-------------------|
| `teacher` | `/teacher/calendar` |
| `student` | `/student/schedule` |

- Неаутентифицированный пользователь при обращении к защищённым маршрутам перенаправляется на `/login`.
- Перекрёстный доступ запрещён: `teacher` не открывает `/student/*`, `student` — не открывает `/teacher/*` (редирект на свою стартовую).

**Форма входа:** идентификатор пользователя — **e-mail** (в UI допускается подпись «Логин» или «Email», значение валидируется как адрес электронной почты); поле **«Пароль»**; переключатель роли «Преподаватель» / «Ученик»; кнопка «Войти»; опционально подсказка с тестовыми кредами только в dev.

## Экран 1 — Вход (`/login`)

### Цель

Вход в систему с выбором роли.

### Роли

- Доступен без аутентификации.

### Компоненты

- Карточка по центру: иконка/логотип, заголовок «Репетитор», подзаголовок «Выберите роль и войдите в систему».
- Toggle двух кнопок роли (иконки: книга / человек); активная — заливка primary, неактивная — outline.
- Поле **e-mail** (логин = email) и «Пароль».
- Кнопка «Войти» (полная ширина, с иконкой входа).
- `ThemeToggle` в углу экрана (вне карточки).

### Действия пользователя

- Выбор роли → ввод **e-mail** и пароля → «Войти» → редирект по роли (см. таблицу выше).

### Edge cases

- Неверные учётные данные — сообщение об ошибке (без раскрытия деталей).
- Пустые поля — валидация на клиенте до запроса.
- Некорректный формат e-mail — валидация на клиенте (и `422` от API при несоответствии схеме).

## Экран 2 — Календарь преподавателя (`/teacher/calendar`)

### Цель

Обзор недельного расписания, быстрые действия по занятиям, обзор активности бота и запросов.

### Роли

- Только `teacher`.

### Компоненты

**Верхняя зона — недельное расписание**

- Навигация: «‹», «Сегодня», «›», отображение диапазона дат недели.
- Сетка из **7 колонок** (ПН–ВС); в заголовке — день недели и число.
- **Текущий календарный день** выделен рамкой (акцентный цвет, напр. оранжевый).
- В колонке дня: карточки занятий **или** текст «Нет занятий».
- Карточка занятия: ФИО ученика, интервал времени, тема; внизу **5 иконок-флагов** (см. легенду).
- Кнопка «+» в колонке дня — создание занятия на этот день.

**Легенда** под сеткой: подписи к иконкам — Уведомление, Подтверждено, ДЗ отправлено, Решение получено, Проверено.

**Нижняя зона — четыре панели (сетка 2×2)**

1. **Лента Telegram-бота (последние 10):** ФИО/фамилия, текст сообщения, время; прокрутка при переполнении.
2. **Запросы на перенос:** карточки с ФИО, предложенным временем, причиной; кнопки «Одобрить» / «Отклонить».
3. **Неподтверждённые (2 дня):** список занятий без подтверждения учеником; кнопка массового напоминания; пустое состояние — например «Все занятия подтверждены».
4. **Несданные ДЗ (2 дня):** список по правилам продукта; кнопка массового напоминания; пустое состояние — например «Все ДЗ получены».

### Действия пользователя

- Переключение недель; переход к «сегодня».
- Создание занятия («+» или из диалога).
- Открытие карточки занятия — редактирование (см. диалог занятия).

### Edge cases

- Пустая неделя — колонки с «Нет занятий».
- Пустые ленты — плейсхолдеры в панелях.

## Диалог создания / редактирования занятия

### Цель

Создать или изменить занятие, управлять флагами и быстрыми действиями.

### Роли

- Только `teacher`.

### Компоненты

- Заголовок: «Создать занятие» / «Редактировать занятие»; кнопка закрытия.
- **Ученик** — выпадающий список.
- **Дата** (date picker), **Начало** и **Конец** (time) — два времени; длительность на backend может храниться как `duration_minutes`, UI считает из интервала.
- **Тема** — однострочное поле.
- Блок **«Статусы»:** 5 переключателей (Switch):
  1. Уведомление отправлено  
  2. Подтверждено учеником  
  3. ДЗ отправлено  
  4. Решение получено  
  5. Решение проверено  

- Кнопки: «Отправить ДЗ», «Прикрепить файл» (заглушка), «Напомнить».
- Блок **«Комментарии»** — поле ввода и добавление комментария (семантика хранения — по согласованию с backend; базовое поле заметок к занятию — см. [data-model.md](../data-model.md) `Lesson.notes`).

- Футер: «Удалить» (только редактирование, destructive), «Отмена», «Сохранить».

### Поведение флагов

- **Переключение каждого флага сохраняется сразу** при переключении: запрос к API обновления флагов **без** нажатия «Сохранить» (optimistic update допустим).

### Действия «Сохранить» / «Удалить»

- «Сохранить» — основные поля формы (ученик, дата/время, тема, комментарий/заметки).
- «Удалить» — только в режиме редактирования, с подтверждением.

### Edge cases

- Конец раньше начала — валидация.
- Ошибка сети при автосохранении флага — откат переключателя и уведомление.

## Экран 3 — Ученики (`/teacher/students`)

### Цель

Список учеников преподавателя, создание и редактирование, переход к детальной карточке.

### Роли

- Только `teacher`.

### Компоненты (список)

- Счётчик «Всего учеников: N».
- Переключатель вида: **карточки** / **таблица** (сохранение выбора — `localStorage` или query).
- Кнопка «+ Добавить».

**Карточка:** ФИО; иконки редактирования и удаления; класс, телефон, e-mail; **Telegram (числовой ID)** при заполнении; разделитель; блок текста **заметок/цели** (например «Подготовка к ОГЭ»).

**Таблица:** колонки ФИО, Класс, Телефон, E-mail, **Telegram**, действия (редактировать / удалить). Поле заметок в таблице не обязательно (достаточно детальной страницы и формы).

### CRUD-диалог ученика

- Заголовок «Редактировать» / «Добавить» (формулировка по согласованию).
- Поля: **ФИО**; в одной строке **Телефон** и **Email**; **Telegram (ID)** — числовой идентификатор пользователя в Telegram (необязательно; пустое значение — нет привязки); **Класс**; **Заметки** (многострочное).
- Кнопки: «Отмена», «Сохранить». Удаление — с карточки/строки, не из диалога.

### Edge cases

- Удаление — подтверждение.
- Пустой список — плейсхолдер.

### Пробел модели данных

- Для текста вроде «Подготовка к ОГЭ» в карточке нужно поле **заметок у ученика** (`users.notes` или аналог) — закрывается в итерации backend ([tasklist-frontend.md](../tasks/tasklist-frontend.md), анализ пробелов схемы).

## Экран 3б — Детальная страница ученика (`/teacher/students/{id}`)

### Цель

Профиль ученика, краткая статистика, история занятий, диалог с ботом.

### Роли

- Только `teacher`.

### Компоненты

- Ссылка «← Назад к списку».
- Заголовок: ФИО + бейдж класса; в блоке контактов при наличии — **Telegram ID**.
- **Четыре карточки метрик**, например: число занятий; число «ДЗ отправлено»; **процент выполнения ДЗ** (процент выделить цветом, напр. зелёным); число «Проверено» (семантика счётчиков — по согласованию с API).

**История занятий** — таблица: дата, время, тема, колонка статусов с теми же **5 иконками**, что в календаре.

**Диалог с ботом** — лента сообщений с временем. Сообщения **`user` и `assistant` различаются визуально** (выравнивание/фон). В тестовых данных могли отсутствовать ответы ассистента — UI должен корректно показывать оба типа.

### Поведение ленты диалога

- Подгрузка старых сообщений при прокрутке вверх (бесконечный скролл), если API отдаёт порциями.

### Edge cases

- Нет занятий / нет сообщений — пустые состояния.

## Экран 4 — Настройки (`/teacher/settings`)

### Цель

Параметры преподавателя и напоминаний.

### Роли

- Только `teacher`.

### Компоненты

Две карточки:

1. **Общие настройки:** «Имя репетитора»; «Длительность занятия (мин)».
2. **Автоматические напоминания:**  
   - «За сколько часов напоминать о занятии» + пояснение про Telegram-бота;  
   - «За сколько часов напоминать о несданном ДЗ» + пояснение.

- Одна основная кнопка **«Сохранить»** (с иконкой дискеты). Отдельная кнопка «Сбросить» **в референсе отсутствует** — опционально в реализации (сброс к последним загруженным с сервера).

### Edge cases

- Валидация числовых полей (минимум/максимум по контракту API).
- Ошибка сохранения — сообщение пользователю.

## Экран 5 — Расписание ученика (`/student/schedule`)

### Цель

Просмотр собственного расписания на неделю.

### Роли

- Только `student`.

### Компоненты

- Заголовок вида «Расписание — {ФИО}».
- Та же **недельная сетка**, что у преподавателя: навигация по неделе, 7 колонок, выделение текущего дня, карточки с временем, темой и **5 флагами**, легенда под сеткой.
- **Без** кнопок добавления занятий; **без** нижних панелей (бот, переносы, напоминания).

### Отличия от tasklist (итерация 3/6)

- В референсе **нет** переключателя «Месяц / Неделя» — только **недельный** вид. Месячный вид — отложенный scope, если не появится в отдельном требовании.

### Edge cases

- День без занятий — «Нет занятий».

## Соответствие vision

| Принцип | Как соблюдено |
|--------|----------------|
| Тонкий клиент | Данные и правила — на backend; UI вызывает API. |
| Роли student / teacher | Разные меню и маршруты; проверка на сервере. |
| LLM только в backend | Диалог на детальной странице — чтение истории из API, без прямых вызовов LLM из браузера. |
