# Техническое видение

Документ описывает целевое устройство проекта Telegram-бота с LLM для поддержки учёбы (математика 7–9, ОГЭ и смежные темы). См. продуктовую идею в [idea.md](idea.md).

---

## Технологии

| Область | Выбор |
|--------|--------|
| Язык | Python 3.12+ (минимальная версия — зафиксировать в `pyproject.toml`) |
| Окружение | `venv` в репозитории (например `.venv/`), не коммитить |
| Зависимости | **uv** — `uv sync` / `uv add`; lock-файл по политике uv |
| Telegram | **aiogram**, режим **long polling** (без webhook на первом этапе) |
| LLM | HTTP-клиент к **OpenAI-compatible** API; базовый URL и ключ — под **OpenRouter** |
| Автоматизация | **GNU Make** (`Makefile`): цели по согласованию — как минимум `install`, `run`; при желании `lint`, `format` |

Зависимости приложения (ориентир): `aiogram`, `openai` (или тонкая обёртка поверх `httpx` — только если появится причина не тянуть официальный SDK), `pydantic` / `pydantic-settings` для конфигурации, при необходимости — лёгкая БД-обёртка (например `aiosqlite`) когда понадобится персистентность.

**Сборка:** не отдельный артефакт; «сборка» = установка зависимостей через uv и проверка, что приложение стартует.

---

## Принципы разработки

- **KISS:** минимум слоёв и абстракций; фичи добавлять по мере необходимости.
- **Один класс — один файл** (где уместно класс); не смешивать несвязанные сущности в одном модуле без причины.
- Явные зависимости (конструкторы, простые фабрики), без тяжёлого DI-фреймворка.
- Конфигурация из окружения + `.env` (локально), секреты не в коде и не в git.
- Типизация и линтер/форматтер — по согласованию в `Makefile` (например Ruff).

## Архитектура

1. **Слой Telegram (aiogram):** роутеры и хендлеры принимают апдейты, парсят команды и текст, вызывают сервисы.
2. **Сервисный слой:** напоминания, планирование, формирование промптов, вызов LLM-клиента, чтение/запись «памяти» ученика.
3. **Интеграция LLM:** один тонкий модуль (класс-клиент в отдельном файле), единая точка вызова чата/комплишенов.
4. **Хранение:** на старте — допустимо in-memory или один файл SQLite; рост — без смены публичного контракта сервисов, где возможно.

Поток: **Update → Handler → Service → (Storage | LLM Client) → ответ пользователю**.

Планировщик напоминаний: проще всего asyncio-задача или периодический опрос из БД в том же процессе, что и polling; отдельный worker — только если появится нагрузка или требования к надёжности.

## Модель данных и состояние

**Сущности (логическая модель, первая итерация):**

- **Ученик** — привязка к `telegram_user_id`, опционально имя/класс для промпта.
- **Занятие** — дата/время, тема, что принести, статус (запланировано / прошло).
- **Домашнее задание** — текст задания, срок сдачи, флаг выполнения/напоминаний.
- **Параметрическая память** — структурированные или текстовые факты о текущем курсе ученика (тема недели, договорённости), которые подмешиваются в контекст LLM.

**Технически:** таблицы в SQLite (или эквивалент) + репозитории в отдельных файлах/классах; миграции — вручную или одним скриптом на ранней стадии, без тяжёлого ORM, если не понадобится.

**Состояние диалога (FSM aiogram):** только там, где нужны пошаговые сценарии (например ввод расписания); остальное — stateless между сообщениями с опорой на БД.

## Работа с LLM

- **Провайдер:** OpenRouter (`OPENROUTER_API_KEY`, базовый URL из документации OpenRouter для OpenAI-compatible API).
- **Клиент:** библиотека с совместимым API (например официальный `openai` с `base_url` на OpenRouter), одна обёртка в проекте для смены модели и таймаутов.
- **Модель:** задаётся конфигом (`LLM_MODEL` или аналог), не захардкожена в коде хендлеров.
- **Промпт:** системная роль (помощник по математике и организации учёбы), пользовательское сообщение, опционально блок «памяти» и краткий контекст занятий/ДЗ из БД.
- **Ограничения:** лимиты токенов и обработка ошибок API (короткое сообщение пользователю + лог).

## Сценарии работы пользователя

Соответствуют [idea.md](idea.md):

- Узнать дату/время и тему **следующего занятия**, что **принести**.
- Получить **напоминания** о занятии и о **сдаче ДЗ**.
- Спросить **как решить** задачу или **объяснить тему** (в рамках роли и без выдуманных требований к «файлам курса»).
- Уточнить, **что задано на дом**.

Реализация: команды и свободный текст; при необходимости — inline-кнопки для частых действий.

## Конфигурация и секреты

- **Обязательно:** `TELEGRAM_BOT_TOKEN`, `OPENROUTER_API_KEY` (или общий ключ для выбранного OpenAI-compatible endpoint).
- **Желательно:** `OPENROUTER_BASE_URL`, `LLM_MODEL`, уровень логов, путь к БД.
- Файл `.env` локально; в репозитории — только `.env.example` с пустыми/фиктивными значениями и комментариями.
- В проде — переменные окружения хостинга, не файл `.env` в образе с секретами.

## Логирование

- Стандартный модуль `logging`; уровень из конфига.
- В логах не писать токены, полные тексты промптов с ПДн — по политике; при отладке — маскирование или отдельный debug-режим локально.
- Структура: префикс с именем модуля, при ошибках LLM/Telegram — `exc_info` где уместно.

## Сборка и запуск

1. `python -m venv .venv` и активация (Windows/Linux — как в README).
2. `uv sync` (или цель `make install`, вызывающая uv).
3. Заполнить `.env` по `.env.example`.
4. `make run` → например `uv run python -m ttlg_bot` (точная команда — в `Makefile`).

**Деплой:** на первом этаце не фиксируется; возможны VPS с systemd, Docker, или PaaS — описать в README позже, когда появится среда.
