# Пользовательские сценарии

Документ фиксирует базовые сценарии взаимодействия ученика и преподавателя с системой — без технических деталей реализации. Для каждого сценария указаны: шаги, ожидаемый результат, канал и задействованные логические сущности из [`docs/data-model.md`](../data-model.md).

Сценарии служат основой для проектирования схемы данных (итерация 2) и разработки API/frontend.

---

## Ученик

### SC-S-01: Узнать дату и тему следующего занятия

**Канал:** Бот, Веб

**Шаги:**
1. Ученик открывает бот или веб-приложение.
2. Запрашивает информацию о следующем занятии (командой, кнопкой или вопросом в свободной форме).
3. Система находит ближайшее запланированное занятие для данного ученика.
4. Ученик получает: дату и время, тему, имя преподавателя, пометку о том, что нужно принести/подготовить (если есть в заметках).

**Ожидаемый результат:** ученик знает, когда и что будет на занятии, без необходимости уточнять у преподавателя.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id`, `name` (преподаватель) |
| `Lesson` | `student_id`, `teacher_id`, `topic`, `scheduled_at`, `status` (`scheduled`), `notes` |

---

### SC-S-02: Получить напоминание о занятии и о ДЗ

**Канал:** Бот

**Шаги:**
1. Система автоматически (по расписанию) проверяет предстоящие занятия и дедлайны ДЗ.
2. За N часов до занятия ученик получает push-уведомление в бот: дата, время, тема.
3. За N часов до дедлайна ДЗ — отдельное уведомление: что задано, срок сдачи, текущий статус.
4. Ученик читает напоминание; при необходимости переходит к деталям.

**Ожидаемый результат:** ученик не пропускает занятия и вовремя сдаёт домашние задания.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id`, `telegram_id` (для отправки уведомления) |
| `Lesson` | `student_id`, `scheduled_at`, `status`, `topic` |
| `Assignment` | `student_id`, `due_date`, `status`, `description` |

---

### SC-S-03: Задать учебный вопрос ассистенту

**Канал:** Бот, Веб

**Шаги:**
1. Ученик пишет вопрос в свободной форме: «Как решить квадратное уравнение?», «Объясни теорему Пифагора».
2. Система определяет текущего ученика по `telegram_id` (бот) или сессии (веб).
3. Backend формирует контекст: тема последнего занятия, активное ДЗ.
4. LLM генерирует ответ с учётом персонального контекста ученика.
5. Ответ сохраняется в историю диалога; ученик видит его в чате.

**Ожидаемый результат:** ученик получает объяснение с учётом своей темы и уровня, без необходимости описывать контекст заново.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id`, `telegram_id`, `role` |
| `Lesson` | `student_id`, `topic`, `scheduled_at` (для контекста) |
| `Dialogue` | `student_id`, `channel`, `started_at` |
| `Message` | `dialogue_id`, `role`, `content`, `created_at` |

---

### SC-S-04: Узнать что задано и статус сдачи ДЗ

**Канал:** Бот, Веб

**Шаги:**
1. Ученик запрашивает список домашних заданий («Что задано?», «Что я уже сдал?»).
2. Система возвращает список актуальных ДЗ: описание, срок, статус (`pending` / `submitted` / `overdue`).
3. При необходимости ученик уточняет детали конкретного задания.

**Ожидаемый результат:** ученик видит полную картину по ДЗ — что нужно сдать, что уже сдал, что просрочено.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id` |
| `Assignment` | `student_id`, `description`, `due_date`, `status`, `lesson_id` |
| `Lesson` | `id`, `topic` (для связки «к какому занятию относится ДЗ») |

---

### SC-S-05: Подтвердить занятие / запросить изменение расписания

**Канал:** Бот

**Шаги:**
1. Ученик получает напоминание о занятии (SC-S-02) с кнопками «Подтверждаю» / «Нужно перенести».
2. При нажатии «Подтверждаю» — система фиксирует подтверждение (или статус остаётся `scheduled`).
3. При нажатии «Нужно перенести» — ученик отправляет сообщение с запросом; бот уведомляет преподавателя.
4. Преподаватель меняет расписание; ученик получает уведомление об обновлении.

**Ожидаемый результат:** ученик может подтвердить участие или инициировать перенос без звонков и переписки.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id`, `telegram_id` |
| `Lesson` | `student_id`, `teacher_id`, `scheduled_at`, `status` |

> **Примечание:** Механизм запроса переноса (отдельная сущность `RescheduleRequest` или поле-флаг в `Lesson`) в текущей схеме не предусмотрен — потенциальный пробел для итерации 2.

---

### SC-S-06: Посмотреть историю занятий и свой прогресс

**Канал:** Веб

**Шаги:**
1. Ученик открывает раздел «Прогресс» в веб-приложении.
2. Система показывает: список пройденных занятий с темами и датами, статистику ДЗ (выполнено / всего), агрегированную оценку за период.
3. При необходимости ученик фильтрует по периоду.
4. Опционально: читает комментарий преподавателя к периоду.

**Ожидаемый результат:** ученик видит свой прогресс в цифрах и понимает, насколько активно занимается.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id` |
| `Lesson` | `student_id`, `status` (`completed`), `scheduled_at`, `topic` |
| `Assignment` | `student_id`, `status`, `due_date` |
| `Progress` | `student_id`, `period_start`, `period_end`, `lessons_completed`, `assignments_done`, `assignments_total`, `summary` |

## Преподаватель

### SC-T-01: Управлять расписанием занятий

**Канал:** Веб

**Шаги:**
1. Преподаватель открывает раздел «Расписание» в веб-приложении.
2. Видит список занятий по всем ученикам (или с фильтром по ученику).
3. Создаёт новое занятие: выбирает ученика, дату/время, тему.
4. При необходимости редактирует существующее занятие (тема, время, статус) или отменяет его.
5. Ученик получает уведомление об изменении (SC-S-05).

**Ожидаемый результат:** преподаватель управляет расписанием всех учеников из единого интерфейса.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id`, `name`, `role` (`student` / `teacher`) |
| `Lesson` | `student_id`, `teacher_id`, `topic`, `scheduled_at`, `status`, `notes` |

---

### SC-T-02: Назначить и редактировать домашнее задание

**Канал:** Веб

**Шаги:**
1. Преподаватель открывает страницу ученика или занятия.
2. Добавляет ДЗ: формулировку, срок сдачи; при необходимости привязывает к конкретному занятию.
3. Ученик получает уведомление о новом ДЗ.
4. При необходимости преподаватель редактирует ДЗ (срок, описание) или меняет статус.

**Ожидаемый результат:** все ДЗ фиксируются в системе и видны ученику; нет «устных» заданий, которые теряются.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id` (ученик) |
| `Lesson` | `id` (привязка ДЗ к занятию) |
| `Assignment` | `lesson_id`, `student_id`, `description`, `due_date`, `status` |

---

### SC-T-03: Отметить прохождение занятия и зафиксировать итог

**Канал:** Веб

**Шаги:**
1. После занятия преподаватель открывает карточку занятия.
2. Меняет статус на `completed`.
3. Вносит заметки: что разобрали, что вызвало затруднения, рекомендации.
4. При необходимости создаёт ДЗ по итогам занятия (переход к SC-T-02).

**Ожидаемый результат:** история занятий актуальна; заметки доступны как контекст для LLM-ассистента и для анализа прогресса.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `Lesson` | `id`, `status` (`completed`), `notes` |

---

### SC-T-04: Посмотреть статус ДЗ и активность по ученику

**Канал:** Веб

**Шаги:**
1. Преподаватель открывает страницу ученика.
2. Видит: список актуальных ДЗ с статусами, сколько пройдено занятий, процент выполнения ДЗ.
3. При необходимости фильтрует по периоду.
4. Читает агрегированный комментарий по ученику за период.
5. Обновляет агрегат прогресса (вручную или автоматически).

**Ожидаемый результат:** преподаватель видит реальную картину активности и успеваемости без ручного подсчёта.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `User` | `id`, `name` |
| `Assignment` | `student_id`, `status`, `due_date` |
| `Lesson` | `student_id`, `status` (`completed`), `scheduled_at` |
| `Progress` | `student_id`, `period_start`, `period_end`, `lessons_completed`, `assignments_done`, `assignments_total`, `summary` |

---

### SC-T-05: Вести базу учебных материалов и тем

**Канал:** Веб

**Шаги:**
1. Преподаватель создаёт или редактирует тему (раздел курса).
2. Прикладывает материалы: файлы, ссылки, текстовые пояснения.
3. При планировании занятия выбирает тему из базы.
4. LLM-ассистент использует материалы темы как дополнительный контекст при ответе ученику.

**Ожидаемый результат:** учебный контент структурирован; преподаватель не описывает тему каждый раз заново, LLM отвечает точнее.

**Задействованные сущности:**

| Сущность | Поля |
|----------|------|
| `Lesson` | `topic` (текстовое поле — неполное решение) |

> **Пробел схемы:** Сущности `Topic` (тема/раздел курса) и `Material` (файл, ссылка, текст) в [`docs/data-model.md`](../data-model.md) **отсутствуют**. В `vision.md` они упомянуты как доменные сущности («Тема», «Материал»), но в текущей схеме не реализованы. Требуют добавления при проектировании в итерации 2.

## Пробелы схемы

Расхождения между сценариями и текущей [`docs/data-model.md`](../data-model.md) — для учёта в итерации 2.

| № | Пробел | Сценарии | Рекомендация |
|---|--------|---------|-------------|
| G-01 | Нет сущности `Topic` (тема / раздел курса) | SC-T-05, SC-S-03 | Добавить `Topic`: `id`, `title`, `description`; связать с `Lesson.topic_id` |
| G-02 | Нет сущности `Material` (файл, ссылка, текст) | SC-T-05 | Добавить `Material`: `id`, `topic_id`, `type` (file/link/text), `content/url` |
| G-03 | Нет механизма запроса переноса занятия | SC-S-05 | Рассмотреть флаг `reschedule_requested` в `Lesson` или отдельную сущность |
| G-04 | `Progress` — агрегат без автоматического пересчёта | SC-S-06, SC-T-04 | Уточнить: пересчитывается вручную или по триггеру; задокументировать в ADR |
| G-05 | `Lesson.notes` — одно текстовое поле для всех заметок | SC-T-03, SC-S-03 | Достаточно для MVP; при росте — отдельная сущность `LessonNote` |

Пробелы G-01 и G-02 подтверждены `vision.md` (сущности «Тема» и «Материал» упомянуты в разделе «Доменные сущности»). Их реализация откладывается до итерации 2.
