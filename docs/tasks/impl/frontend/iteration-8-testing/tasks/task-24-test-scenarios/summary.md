# Задача 24 — Тест-сценарии для 5 экранов: summary

Ручные чек-листы для приёмки веб-интерфейса. Каждый сценарий: **предусловие** → **шаги** → **ожидаемый результат**.

---

## Экран 1 — Вход (`/login`)

### Сценарий 1.1 — Успешный вход преподавателя

| | |
|--|--|
| **Предусловие** | Запущены backend и frontend (или стенд); в БД есть пользователь с ролью `teacher`, известные e-mail и пароль. |
| **Шаги** | 1. Открыть `/login`. 2. Убедиться, что выбрана роль «Преподаватель». 3. Ввести e-mail и пароль. 4. Нажать «Войти». |
| **Ожидаемый результат** | Редирект на `/teacher/calendar`; в шапке/sidebar доступны пункты преподавателя; сессия сохранена (обновление страницы не выкидывает на логин). |

### Сценарий 1.2 — Успешный вход ученика

| | |
|--|--|
| **Предусловие** | В БД есть пользователь с ролью `student`, известные e-mail и пароль. |
| **Шаги** | 1. Открыть `/login`. 2. Выбрать роль «Ученик». 3. Ввести e-mail и пароль. 4. Нажать «Войти». |
| **Ожидаемый результат** | Редирект на `/student/schedule`; в навигации только «Моё расписание» и «Выйти». |

### Сценарий 1.3 — Ошибочный пароль

| | |
|--|--|
| **Предусловие** | Пользователь с известным e-mail существует. |
| **Шаги** | 1. Выбрать корректную роль. 2. Ввести e-mail и **неверный** пароль. 3. Нажать «Войти». |
| **Ожидаемый результат** | Сообщение об ошибке (например «Неверный логин или пароль»); остаёмся на `/login`; пароль не подсвечивает детали с сервера. |

### Сценарий 1.4 — Редирект по роли (перекрёстный доступ)

| | |
|--|--|
| **Предусловие** | Выполнен вход как `teacher`. |
| **Шаги** | 1. Вручную открыть URL `/student/schedule`. |
| **Ожидаемый результат** | Редирект на стартовую страницу преподавателя (`/teacher/calendar` или эквивалент по проекту). |

| | |
|--|--|
| **Предусловие** | Выполнен вход как `student`. |
| **Шаги** | 1. Вручную открыть `/teacher/calendar`. |
| **Ожидаемый результат** | Редирект на `/student/schedule`. |

### Сценарий 1.5 — Валидация на клиенте

| | |
|--|--|
| **Предусловие** | Страница `/login` открыта. |
| **Шаги** | 1. Оставить поля пустыми и нажать «Войти» **или** 2. Ввести некорректный e-mail без `@` и отправить форму. |
| **Ожидаемый результат** | Браузер/HTML5 или UI блокирует отправку / показывает ошибку; запрос с пустыми полями не уходит (или возвращается клиентская ошибка). |

### Сценарий 1.6 — Переключение темы на экране входа

| | |
|--|--|
| **Предусловие** | `/login` открыт. |
| **Шаги** | 1. Нажать переключатель темы в углу экрана несколько раз. |
| **Ожидаемый результат** | Тема переключается light/dark без ошибок; после перезагрузки тема сохраняется (если настроено через `next-themes`). |

## Экран 2 — Календарь преподавателя (`/teacher/calendar`)

### Сценарий 2.1 — Навигация по неделям

| | |
|--|--|
| **Предусловие** | Вход как `teacher`, есть хотя бы одно занятие в соседней неделе (или пустая неделя). |
| **Шаги** | 1. Нажать «›» (следующая неделя). 2. Нажать «‹». 3. Нажать «Сегодня». |
| **Ожидаемый результат** | Диапазон дат в заголовке меняется корректно; сетка из 7 колонок (ПН–ВС); текущий календарный день визуально выделен. |

### Сценарий 2.2 — CRUD занятия (создание и редактирование)

| | |
|--|--|
| **Предусловие** | Вход как `teacher`; есть хотя бы один ученик в системе. |
| **Шаги** | 1. В колонке дня нажать «+ Занятие» (или аналог). 2. Заполнить ученика, дату/время, тему; сохранить. 3. Кликнуть по карточке занятия; изменить тему или время; сохранить. |
| **Ожидаемый результат** | Новое занятие появляется в сетке; после редактирования данные обновляются без перезагрузки страницы (или после revalidate). |

### Сценарий 2.3 — Удаление занятия

| | |
|--|--|
| **Предусловие** | Есть занятие в текущей неделе. |
| **Шаги** | 1. Открыть диалог редактирования. 2. Нажать «Удалить», подтвердить. |
| **Ожидаемый результат** | Карточка исчезает из сетки; в этот день при отсутствии других занятий — «Нет занятий» или аналог. |

### Сценарий 2.4 — Пять флагов на карточке и мгновенное сохранение

| | |
|--|--|
| **Предусловие** | Открыто занятие в диалоге или на карточке есть переключатели флагов (по ТЗ — переключение сразу сохраняется). |
| **Шаги** | 1. По очереди переключить каждый из 5 флагов (уведомление, подтверждено, ДЗ, решение, проверено). 2. Обновить страницу. |
| **Ожидаемый результат** | На карточке видны 5 иконок/индикаторов; после переключения состояние сохраняется на сервере; после F5 состояние совпадает. При ошибке сети — откат переключателя и уведомление (если реализовано). |

### Сценарий 2.5 — Кнопки напоминаний

| | |
|--|--|
| **Предусловие** | В панели «Неподтверждённые» / «Несданные ДЗ» есть элементы или пустое состояние. |
| **Шаги** | 1. Нажать массовое напоминание (если список не пуст). 2. Проверить пустое состояние (все подтверждено / все ДЗ сданы). |
| **Ожидаемый результат** | Запрос уходит без ошибки; при успехе — обратная связь (toast/счётчик); пустые панели показывают дружелюбный текст. |

### Сценарий 2.6 — Лента бота и запросы на перенос

| | |
|--|--|
| **Предусловие** | Вход как `teacher`; при наличии тестовых данных — есть сообщения бота и запросы переноса. |
| **Шаги** | 1. Прокрутить ленту последних сообщений. 2. Для запроса переноса нажать «Одобрить» или «Отклонить». |
| **Ожидаемый результат** | Сообщения отображаются с ФИО и временем; после действия карточка запроса обновляется или исчезает; ошибка API показывается пользователю. |

## Экран 3 — Ученики (`/teacher/students` и `/teacher/students/{id}`)

### Сценарий 3.1 — Переключение вида списка

| | |
|--|--|
| **Предусловие** | `/teacher/students` открыт, есть ≥1 ученик. |
| **Шаги** | 1. Переключить «карточки» / «таблица». 2. Обновить страницу. |
| **Ожидаемый результат** | Вид меняется; выбор сохраняется (localStorage или query — по реализации). |

### Сценарий 3.2 — CRUD ученика (создание)

| | |
|--|--|
| **Предусловие** | Вход как `teacher`. |
| **Шаги** | 1. Нажать «Добавить». 2. Заполнить ФИО, опционально класс, телефон, email, Telegram ID, заметки. 3. Сохранить. |
| **Ожидаемый результат** | Новый ученик в списке; счётчик «Всего учеников» увеличился. |

### Сценарий 3.3 — Редактирование и удаление

| | |
|--|--|
| **Предусловие** | Есть ученик в списке. |
| **Шаги** | 1. Открыть редактирование с карточки/таблицы; изменить поле; сохранить. 2. Нажать удалить; подтвердить в диалоге. |
| **Ожидаемый результат** | Данные обновлены в списке; после удаления ученик исчезает; при конфликте `telegram_id`/`email` — сообщение об ошибке (409). |

### Сценарий 3.4 — Детальная страница: метрики, история, диалог

| | |
|--|--|
| **Предусловие** | Есть ученик с историей занятий и/или сообщениями диалога. |
| **Шаги** | 1. Перейти по ссылке с карточки на `/teacher/students/{id}`. 2. Проверить 4 карточки метрик. 3. Прокрутить таблицу занятий (5 флагов). 4. Прокрутить ленту диалога (user vs assistant). |
| **Ожидаемый результат** | Метрики соответствуют API; таблица занятий согласована с календарём; сообщения разных ролей визуально различимы; пустые состояния корректны. |

### Сценарий 3.5 — Бесконечная подгрузка диалога (если включена)

| | |
|--|--|
| **Предусловие** | У ученика много сообщений в диалоге. |
| **Шаги** | 1. Прокрутить ленту вверх до конца. |
| **Ожидаемый результат** | Подгружаются более старые сообщения без дублирования и сбоев. |

## Экран 4 — Настройки (`/teacher/settings`)

### Сценарий 4.1 — Загрузка и отображение формы

| | |
|--|--|
| **Предусловие** | Вход как `teacher`; в API есть сохранённые настройки. |
| **Шаги** | 1. Открыть `/teacher/settings`. |
| **Ожидаемый результат** | Поля «Имя репетитора», длительность занятия, часы напоминаний заполнены значениями с сервера. |

### Сценарий 4.2 — Сохранение

| | |
|--|--|
| **Предусловие** | Форма открыта. |
| **Шаги** | 1. Изменить одно или несколько полей. 2. Нажать «Сохранить». |
| **Ожидаемый результат** | Успешное уведомление; после F5 значения сохранены; ошибка API показывается текстом/toast. |

### Сценарий 4.3 — Сброс

| | |
|--|--|
| **Предусловие** | Форма открыта; поля изменены локально. |
| **Шаги** | 1. Нажать «Сбросить». |
| **Ожидаемый результат** | Поля возвращаются к **последним загруженным с сервера** (baseline после последнего успешного сохранения). |

### Сценарий 4.4 — Валидация

| | |
|--|--|
| **Предусловие** | Форма открыта. |
| **Шаги** | 1. Очистить «Имя репетитора» или ввести длительность вне диапазона (например &lt; 15). 2. Нажать «Сохранить». |
| **Ожидаемый результат** | Подсветка ошибок у полей; запрос на сервер не уходит или сервер отклоняет с понятным сообщением. |

## Экран 5 — Расписание ученика (`/student/schedule`)

### Сценарий 5.1 — Недельная сетка и навигация

| | |
|--|--|
| **Предусловие** | Вход как `student`. |
| **Шаги** | 1. Проверить заголовок с ФИО. 2. Листать недели «‹» / «›» / «Сегодня». |
| **Ожидаемый результат** | Сетка 7 колонок; текущий день выделен; отображается только расписание текущего пользователя. |

### Сценарий 5.2 — Карточка занятия (только просмотр)

| | |
|--|--|
| **Предусловие** | У ученика есть занятия на неделе. |
| **Шаги** | 1. Найти карточку занятия. 2. Проверить время, тему, 5 флагов. |
| **Ожидаемый результат** | Данные совпадают с API; **нет** кнопки «+ Занятие»; клик по карточке не открывает редактирование (если не задумано иначе). |

### Сценарий 5.3 — Пустой день

| | |
|--|--|
| **Предусловие** | Есть день без занятий в выбранной неделе. |
| **Шаги** | 1. Перейти на такую неделю. |
| **Ожидаемый результат** | В колонке дня текст вроде «Нет занятий». |

### Сценарий 5.4 — Нет нижних панелей преподавателя

| | |
|--|--|
| **Предусловие** | Вход как `student`, страница расписания открыта. |
| **Шаги** | 1. Прокрутить страницу вниз. |
| **Ожидаемый результат** | Нет блоков «бот», «переносы», «неподтверждённые», «несданные ДЗ» как на календаре учителя. |

## Рекомендуемые «ключевые» сценарии для быстрой приёмки (≥5)

1. 1.1 — вход teacher  
2. 1.2 — вход student  
3. 2.1 + 2.2 — календарь + создание занятия  
4. 3.2 или 3.4 — ученики / детальная  
5. 4.2 — настройки сохранение  
6. 5.2 — расписание ученика (read-only)

### Приёмка пользователя

Ключевые сценарии из списка выше пройдены вручную (2026-04-22).
