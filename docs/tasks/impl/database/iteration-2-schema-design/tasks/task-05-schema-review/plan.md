## Заранее известные точки проверки

Ниже — темы, выявленные при анализе схемы до применения skill. Для каждой нужно либо подтвердить, либо пересмотреть по итогам чек-листа.

| # | Тема | Вопрос |
|---|------|--------|
| R-01 | `TIMESTAMPTZ` | Все datetime-поля используют `DateTime(timezone=True)` — подтвердить соответствие best practice |
| R-02 | UUID PK | Все PK — `UUID` без serial/sequence — подтвердить, зафиксировать обоснование |
| R-03 | Составной UNIQUE на `progress` | Стоит ли добавить `UNIQUE(student_id, period_start, period_end)`? Предотвращает дублирование записей за один период |
| R-04 | Составной индекс на `messages` | Стоит ли добавить `(dialogue_id, created_at)` вместо простого `dialogue_id`? Полезен для выборки сообщений диалога в порядке времени |
| R-05 | Индекс на `lessons.scheduled_at` | Нужен для сценария SC-S-01 («следующее занятие»): `WHERE student_id = ? AND scheduled_at > now() ORDER BY scheduled_at LIMIT 1` |
| R-06 | `Assignment.status = overdue` | Значение `overdue` не вычисляется автоматически — обновляется вручную или фоновой задачей; нужно зафиксировать подход в документации |
| R-07 | Отсутствие `updated_at` | `Lesson` и `Assignment` могут меняться (статус, заметки, дедлайн); оценить необходимость поля `updated_at` |
| R-08 | NULL-семантика FK | `Assignment.lesson_id` nullable (SET NULL) — корректна ли семантика «ДЗ без занятия»? |
| R-09 | Именование enum-типов | `user_role`, `lesson_status`, `assignment_status`, `dialogue_channel`, `message_role` — проверить соответствие PostgreSQL-конвенциям |
| R-10 | Нормализация `Progress` | `Progress` — агрегат с дублируемыми данными; подтвердить, что денормализация осознанна |
