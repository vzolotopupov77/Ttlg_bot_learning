## Цель

Актуализировать логическую модель в [`docs/data-model.md`](../../../../data-model.md) с учётом сценариев из итерации 1; нарисовать физическую ER-диаграмму с PostgreSQL-типами, constraints и индексами; провести ревью схемы через skill `postgresql-table-design` и зафиксировать принятые решения.

## Ценность

После итерации есть единый актуальный источник правды по схеме, пригодный для ревью кода, онбординга и работы с frontend-командой.

- [`docs/data-model.md`](../../../../data-model.md) — текущая документация (устарела: `UUID / int` вместо `UUID`, нет каскадов и индексов)

### Зафиксированные расхождения models.py ↔ data-model.md

| Расхождение | В коде | В документе |
|-------------|--------|-------------|
| PK всех сущностей | `UUID` | `UUID / int` |
| `User.telegram_id` | `BIGINT, UNIQUE, nullable` | `int (nullable)` |
| Timestamps | `DateTime(timezone=True)` → `TIMESTAMPTZ` | `datetime` |
| Длины строк | `VARCHAR(255)` для `name`, `VARCHAR(512)` для `topic` | не указаны |
| FK каскады | `CASCADE` / `SET NULL` | не задокументированы |
| Индексы | `index=True` на всех FK-колонках | не задокументированы |

## Ограничения

- Изменения кода (`models.py`, миграции) в этой итерации не реализуются — только документирование. Если ревью выявит необходимые изменения ORM — они помечаются для задачи 11 итерации 5.
- Задача 03 должна быть завершена до начала задачи 04 (физическая ERD строится на основе актуализированной логической модели).
