# ADR-003: Инструменты качества кода

| | |
|---|---|
| **Статус** | Принято |
| **Дата** | 2026-04-01 |
| **Контекст** | Монорепо с двумя Python-пакетами (`ttlg-bot`, `ttlg-backend`); нужны единые линт, форматирование и управление dev-зависимостями без расхождений между пакетами. |

---

## Контекст

После реализации этапов 1–7 (core backend, диалог, документация) сформировался монорепо на **uv workspace**. Нет единого лinter/formatter; dev-зависимости тестов (`pytest`, `httpx` и др.) объявлены в `[project.optional-dependencies]`, что в uv-workspace не устанавливается автоматически при `uv sync`. Нужно:

- единый инструмент линтинга и форматирования для обоих пакетов;
- воспроизводимая установка dev-инструментов (одна команда → рабочая среда);
- минимум команд для проверки перед PR.

---

## Рассмотренные варианты

### Линтер + форматтер

#### 1. Ruff

**Плюсы:**
- Один инструмент вместо связки flake8 + black + isort + pyupgrade; порядки быстрее.
- Конфиг в `pyproject.toml`; охват нескольких src-деревьев через `src = [...]`.
- Активно поддерживается Astral (авторы uv); единый экосистемный стек.

**Минусы:**
- Не все правила flake8-плагинов реализованы; некоторые проверки специфичных плагинов отсутствуют.

**Вывод:** оптимальный выбор для нашего стека.

#### 2. flake8 + black + isort

**Плюсы:** зрелая экосистема, большое число плагинов.

**Минусы:** три инструмента, три конфига, несогласованность версий; ощутимо медленнее.

**Вывод:** избыточно; Ruff перекрывает нужные правила.

#### 3. pylint

**Плюсы:** очень широкий охват проверок.

**Минусы:** медленный; много ложных срабатываний; сложная настройка для проекта с async.

**Вывод:** не выбираем.

---

### Тайпчекер

#### 1. Не вводить сейчас

**Плюсы:** нет накладных расходов на аннотации и конфигурацию на этапе учебного MVP.

**Минусы:** снижает уверенность при рефакторинге.

**Вывод:** откладывается; вернуться при росте команды или переходе к более сложным доменным преобразованиям.

#### 2. ty (Astral)

**Плюсы:** быстрый, тот же стек что у uv/ruff; PEP 561-совместимый.

**Минусы:** проект молодой (0.x); API нестабильный.

**Вывод:** кандидат для ADR-004 по мере зрелости инструмента.

#### 3. mypy

**Плюсы:** де-факто стандарт, огромная экосистема плагинов.

**Минусы:** медленнее ty; для async-SQLAlchemy и pydantic-settings нужна дополнительная настройка.

**Вывод:** не вводим сейчас; возможна замена на ty через отдельный ADR.

---

### Управление dev-зависимостями

#### 1. `[dependency-groups]` (PEP 735) + `[tool.uv] default-groups`

**Плюсы:**
- Современный стандарт; поддержан uv с версии 0.4+.
- `default-groups = ["dev"]` — одна команда `uv sync` даёт рабочую среду разработчика без `--extra`.
- Единая секция для всех dev-инструментов (ruff, pytest, httpx, aiosqlite, respx).

**Минусы:** `[dependency-groups]` не часть core `pyproject.toml` PEP 517/518; часть инструментов ещё не умеет его читать (не актуально: нам нужен только uv).

**Вывод:** использовать.

#### 2. `[project.optional-dependencies]`

**Плюсы:** широко поддержан (pip, pipx, hatch).

**Минусы:** в uv workspace `--extra dev` нужно явно указывать в каждой команде; риск забыть при onboarding.

**Вывод:** заменяем на `[dependency-groups]`.

---

### Makefile-цели

Принятый набор: `lint`, `format`, `check` (= lint + backend-test + bot-test). Минимально необходимый; при появлении тайпчекера — добавить `typecheck` и включить в `check`.

---

## Решение

1. **Линтер + форматтер:** **Ruff**; конфиг в корневом `pyproject.toml`; `target-version = "py312"`, `line-length = 120`; правила `E, F, I, UP, B, SIM`; Alembic-файлы — игнор `E501, E402`.
2. **Тайпчекер:** не вводится сейчас; зафиксировать пересмотр при появлении нового пакета или роста команды (кандидат — **ty**).
3. **Dev-зависимости:** **`[dependency-groups] dev`** в обоих `pyproject.toml`; **`[tool.uv] default-groups = ["dev"]`** в корневом — разработчик получает ruff, pytest, httpx и др. после `uv sync`.
4. **Makefile:** `make lint`, `make format`, `make check`; перед PR — минимум `make check`.

---

## Последствия

- Все dev-пакеты (ruff, pytest, httpx, aiosqlite, respx) — только в `[dependency-groups]`; `[project.optional-dependencies]` не используется для dev-инструментов.
- При добавлении нового пакета в монорепо — сразу объявлять его dev-зависимости в `[dependency-groups]` того же `pyproject.toml`.
- При введении тайпчекера — новый ADR-004; конфиг ty или mypy добавить в корневой `pyproject.toml`; цель `typecheck` и включение в `check` — в Makefile.
- Документ [docs/vision.md](../vision.md) раздел «Принципы разработки» уже содержит ссылку на Ruff — расхождений нет.
