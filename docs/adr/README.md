# Architecture Decision Records (ADR)

Журнал архитектурных решений проекта. Каждое значимое решение фиксируется отдельным файлом.

---

## Зачем

Архитектурные решения — это выбор между альтернативами, у которого есть контекст и последствия. ADR сохраняют этот контекст: почему выбрали именно так, что отвергли и на каких условиях решение может измениться.

---

## Принципы

- **Одно решение — один файл.** Не объединять несвязанные решения.
- **Фиксировать альтернативы.** Документ без них — не ADR, а постфактум-объяснение.
- **Статус обязателен.** Решение живёт в одном из состояний:
  - `Предложено` — обсуждается, не принято.
  - `Принято` — действует.
  - `Отменено` — заменено другим решением (с ссылкой).
  - `Устарело` — контекст изменился, решение неактуально.
- **Не удалять.** Устаревшие ADR остаются — они часть истории проекта.
- **Обновлять при смене решения** — менять статус и добавлять ссылку на новый ADR.

---

## Формат файла

```
adr-NNN-тема.md
```

Заголовок, таблица с метаданными (статус, дата, контекст), разделы:
- Контекст
- Рассмотренные варианты
- Решение
- Последствия
