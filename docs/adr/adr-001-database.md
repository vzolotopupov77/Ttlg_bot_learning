# ADR-001: Выбор СУБД

| | |
|---|---|
| **Статус** | Принято |
| **Дата** | 2026-03-27 |
| **Контекст** | Выбор основной СУБД для backend системы сопровождения учебного процесса |

---

## Контекст

Система состоит из нескольких компонентов: Telegram-бот, backend (API), веб-приложение. Все они обращаются к единому слою данных. Нужна СУБД, которая:

- работает при одновременном обращении из нескольких процессов/сервисов;
- хранит реляционные данные (пользователи, занятия, ДЗ, диалоги);
- не потребует замены при росте системы;
- легко поднимается в dev и prod окружении.

---

## Рассмотренные варианты

### 1. SQLite

**Плюсы:**
- Нулевая инфраструктура: файл на диске.
- Удобен для прототипа с одним процессом.

**Минусы:**
- Не поддерживает конкурентную запись из нескольких процессов — критично уже на старте (bot + backend в разных процессах).
- При деплое в контейнерах: проблема с монтированием файла, потеря данных при пересоздании контейнера.
- Неизбежная миграция при росте — лишние затраты позже.

**Вывод:** подходит только для однопроцессного MVP без продолжения. Наш случай — не тот.

---

### 2. PostgreSQL ✅ — выбрано

**Плюсы:**
- Полноценная конкурентная СУБД: несколько сервисов пишут и читают без конфликтов.
- Богатая типизация: `enum`, `jsonb`, `timestamptz`, массивы.
- Отличная поддержка в Python-стеке: `asyncpg`, `SQLAlchemy`, `Alembic`.
- Поднимается за одну строку: `docker compose up db`.
- Расширяемость без смены СУБД: `pgvector` для embeddings, full-text search — когда понадобится.
- Промышленный стандарт: документация, поддержка, хостинги (Railway, Supabase, Render и др.).

**Минусы:**
- Требует запущенного сервера (в отличие от SQLite).
- Немного больше настройки на старте.

**Вывод:** единственный разумный выбор для многокомпонентной системы с прицелом на рост.

---

### 3. MySQL / MariaDB

**Плюсы:**
- Распространён, дешёвые хостинги.

**Минусы:**
- Хуже поддержка современных типов (`jsonb`, `array`) по сравнению с PostgreSQL.
- Нет экосистемы расширений уровня `pgvector`.
- Python-экосистема (FastAPI, SQLAlchemy async) ориентирована на PostgreSQL.

**Вывод:** не даёт преимуществ перед PostgreSQL; от него отказались.

---

## Решение

**Использовать PostgreSQL** как основную СУБД с первого дня.

Схема управляется через **Alembic** (миграции). ORM — **SQLAlchemy** (async) или лёгкий query-builder — по согласованию при старте backend.

Локально — PostgreSQL в Docker Compose. Prod — managed-инстанс (Railway / Supabase / VPS) или тот же Docker.

---

## Последствия

- Нужен `docker-compose.yml` с сервисом `db` для локальной разработки.
- `.env` содержит `DATABASE_URL` (формат: `postgresql+asyncpg://...`).
- При появлении векторного поиска (RAG, Knowledge Base) — подключить `pgvector` без смены СУБД.
- Redis рассматривается отдельно — только если появится нужда в кеше или очередях (не сейчас).
