# Subagent: `docs-updater`

Контекст для **Task (subagent)** / ручного вызова: актуализация документации при изменениях **публичного HTTP API** в backend. Синхронизируется с [api-conventions.md](../api-conventions.md) и [data-model.md](../data-model.md).

**Указатель для IDE:** [`.cursor/agents/docs-updater.md`](../../.cursor/agents/docs-updater.md) (frontmatter + ссылка сюда).

---

## Назначение

После или вместе с PR, который меняет маршруты FastAPI, Pydantic-схемы ответов, коды ошибок, авторизацию или smoke-сценарии, привести в соответствие **человекочитаемую** спецификацию и артефакты в `docs/`.

---

## Триггеры (когда применять)

- Правки в `backend/src/ttlg_backend/api/**`, `backend/src/ttlg_backend/**` роутеров, зависимостей, `main.py` (префиксы, CORS, `/health`, `/docs`).
- Новые/изменённые Pydantic-модели, участвующие в **публичном** API.
- Изменения в интеграционных/контрактных тестах, явно фиксирующие контракт (если дифф касается публичного поведения).
- Пользователь явно просит «обновить api-contracts / доку по API».

**Не** запускать «на автомате» по любому касанию `backend/`: сначала оценить, **видел ли** это клиент (бот, Next.js) или **дока/Swagger**.

---

## Действия (порядок)

1. **Быстрый дифф по смыслу:** что изменилось в URL, методе, теле, статусах, `error.code`, заголовках, ролях.
2. **OpenAPI-артефакт:** при любом существенном изменении публичного API выполнить в корне репозитория:
   ```bash
   make openapi-export
   ```
   Убедиться, что [openapi.json](../openapi.json) обновлён и коммитится **в одном PR** с кодом/докой (см. [contributing.md](../contributing.md)).
3. **Главный файл контрактов — [docs/tech/api-contracts.md](../tech/api-contracts.md):**
   - таблица **«Обзор маршрутов»** (метод, путь, код успеха, назначение);
   - детальные разделы по затронутым эндпоинтам (тела, enum, примеры ошибок, предусловия);
   - перекрёстные ссылки на [data-model.md](../data-model.md) / [integrations.md](../integrations.md) при смене сущностей или smoke-шагов.
4. **Onboarding — [onboarding.md](../onboarding.md):** обновлять **только если** меняется то, с чем сталкивается новичок: URL для проверок, **новые** обязательные env, порядок smoke-команд, явные критерии «успех» (например новый health-путь). Тонкие детали API — в `api-contracts`, не дублировать полотна.
5. **По ситуации:** [integrations.md](../integrations.md) (curl/smoke), **корневой [README.md](../../README.md)** (если менялся порт, короткий E2E, `make` цели, env), [doc-audit.md](../doc-audit.md) (если добавлен/удалён крупный документ).

---

## Критерии готовности

- Описание в `api-contracts.md` **не противоречит** актуальному `docs/openapi.json` и [api-conventions.md](../api-conventions.md).
- Нет битых внутренних ссылок в изменённых разделах.
- Упоминание `make openapi-export` в PR при изменении схемы (если применяли экспорт в этом PR).

---

## Взаимодействие с кодом

- **Источник правды для полей** — Pydantic-модели и OpenAPI; `api-contracts` даёт сценарии и пояснения, не копипаста всей схемы.
- При сомнении открыть **Swagger** факт: `http://127.0.0.1:8000/docs` (локально) или сгенерированный `openapi.json`.

---

## Ограничения

- Не менять продуктовую **vision** / **tasklist** без отдельной задачи.
- Не расширять `onboarding` новыми разделами «про весь backend» — только релевантные шаги для нового участника.
